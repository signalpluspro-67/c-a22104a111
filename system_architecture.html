· The seven layers

Everything is a pipe. Market data enters at the top, gets stored once in a canonical SQLite hub, gets transformed by a library of pure modules, screened into signals, scheduled by cron, and finally rendered onto surfaces you actually read. Validation cuts across all of it.

The one rule that explains everything: the VPS is canonical, the Mac is a read-only mirror. The VPS runs the pipeline, owns the database, and publishes to GitHub Pages on its own — the Mac can be asleep or off. Every "stale dashboard" and "corrupted DB" bug traces back to something on the Mac fighting that rule.

4 · Screeners — signals into the archive

~40 screeners run each cycle. Every one writes rows to the central signal_archive table (date, screener, ticker, score 0–100, direction, extra_json). The core directional set:

blowoff_screener(archive)SHORT — parabolic exhaustion
signal_archive is polluted by design. Screeners archive WATCH-level rows from score 50. Real triggers start at score ≥ 65 — always filter score >= 65 before any cross-screener confluence ranking, or you rank noise. If BOTTOM + BLOWOFF + PULLBACK all fire on the same name, that's a contradiction (a threshold bug), not confluence.

7 · Validation — the gauntlet across it all

No signal earns size without surviving the 8-step doctrine. The decisive gate is the block bootstrap — it exposes "edges" that were really just a few winning days clustered together. This is what keeps the live system thin and honest. (Full scoreboard in How the system works.)
┌─ THE 8-STEP GAUNTLET ─────────────────────────────────────────┐
│ 1. Sign-flip permutation x5000   (is the mean return real?)   │
│ 2. Bonferroni correction         (penalize many tests)        │
│ 3. Block bootstrap, len>=horizon (edge, or clustered luck?)   │
│ 4. Deflated Sharpe ratio         (correct for # of trials)    │
│ 5. Walk-forward OOS              (does train carry to test?)  │
│ 6. Crash cross-check             (did it warn before drops?)  │
│ 7. Verdict: CONFIRMED / MARGINAL / REJECTED                   │
│                                                               │
│  -> fail one gate early and the signal is out                 │
└───────────────────────────────────────────────────────────────┘
The recurring trap: Bonferroni-OK but block-bootstrap-KO = temporal clustering, not a stationary edge. Mechanical confluence makes the apparent Sharpe rise while the proof collapses (rare = a handful of glued-together episodes). Live truth comes from signal_outcome.db (forward P&L) and screener_validity — re-checked, never assumed.

Four homes, one direction of flow. The Mac develops, the VPS runs and publishes, GitHub Pages serves the public hub, Google Sheets feeds Streamlit, Telegram delivers.

Direction is one-way. deploy_screeners.sh pushes Mac → VPS; sync_db_from_vps.sh pulls the database VPS → Mac. The Mac is never the source of truth for data. For a trade decision, query the VPS DB directly (single writer, always fresh).
